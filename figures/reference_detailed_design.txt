1. 引言

1.1 编写目的
  本文档是在系统需求分析的基础上，对多模态智能模拟面试系统进行详细设
计和说明。主要解决实现该系统需求的程序模块设计问题，包括系统的整体架构、
各模块的功能划分与接口设计、数据处理流程以及关键技术实现方案等。本文档
旨在推动项目开发的规范化，使设计人员和开发人员遵循统一的设计规范，降低
系统实现的风险，保证系统设计的全面性与合理性，为后续的系统实现、测试、
维护和升级提供可靠依据。本文档主要供系统设计人员、开发人员以及测试人员
参考。

1.2 项目背景
  随着人工智能技术的快速发展，智能化面试辅助系统逐渐成为就业市场的重
要工具。传统的面试准备方式往往缺乏实时反馈和专业指导，无法全面评估求职
者的表现。多模态智能模拟面试系统通过结合语音识别、自然语言处理、光学字
符识别(OCR)和大语言模型等多种技术，为求职者提供逼真的面试模拟环境。系
统能够根据不同岗位需求生成专业的面试题目，分析用户的语音、语言表达和行
为举止，提供详细的评估和改进建议。这不仅帮助求职者更好地准备面试，提高
面试通过率，也为招聘方提供更加客观、全面的评估工具，提升招聘效率和质量。
在人才竞争日益激烈的今天，此系统的开发将对就业市场的良性发展起到积极促
进作用。

2. 总体设计

2.1 需求规定

 2.1.1 对功能的规定
  面试模式管理。支持文字面试、语音面试、视频面试三种模式，用户可根据
自身需求选择不同的面试方式。系统能够针对不同面试模式自动适配相应的评估
标准与分析维度。
  题库与岗位管理。能够对各类职位的面试题目进行分类管理，支持根据岗位
类型自动生成针对性的面试题目。能够根据岗位需求调整面试题目数量及难度等
级，确保面试题目的专业性与针对性。
  面试评估。支持对面试者表现的多维度评估，包括专业知识、技能匹配、表
达能力、逻辑思维、创新能力、抗压能力等六个核心维度的量化评分。提供详细
的评价报告，包含总体评价、优势分析、不足指出和改进建议等内容。
  简历分析。支持用户上传个人简历（PDF、Word 等格式），系统通过 OCR
技术提取简历内容，并对简历进行智能分析，生成简历评估报告，为面试评估提
供参考依据。
  资源推荐。根据面试评估结果，针对性地推荐学习资源和提升方案，帮助用
户弥补自身短板，提升面试表现。
  历史记录管理。支持用户查看历史面试记录、评估结果及进步趋势，帮助用
户了解自身能力提升情况。支持对历史面试视频、音频的回放功能，便于用户自
我分析与改进。
  语音交互。支持语音合成功能，将面试题目转换为语音播放，提供更真实的
面试体验；支持语音识别功能，将用户回答内容转换为文本进行分析。
  视频分析。支持对用户面部表情、视线方向、肢体语言等非语言因素进行分
析评估，提供更全面的面试表现评价。

 2.1.2 对性能的规定
     响应时间。系统对绝大多数功能的响应时间控制在 3 秒内，语音识别与转换
的延迟不超过 1 秒，视频分析结果生成时间不超过 10 秒，面试评估报告生成时
间不超过 30 秒。
     并发性能。系统应支持至少 100 名用户同时在线进行模拟面试，确保在高峰
期系统运行稳定。
     可用性。系统服务可用性应达到 99.5%以上，保证用户在需要时能够顺利使
用系统功能。
     准确性。语音识别准确率不低于 95%，面部表情识别准确率不低于 90%，
评估结果与专业人力资源评估的一致性不低于 85%。
     可扩展性。系统采用模块化设计，支持功能扩展和升级，便于未来增加新的
面试模式、评估维度或对接其他 AI 模型。

 2.1.3 对兼容性的规定
     平台兼容性。前端系统支持在主流浏览器（Chrome、Firefox、Safari、Edge

脑和智能手机等多种终端设备上使用。
     数据格式兼容性。支持导入常见的简历文件格式（ PDF 、 DOC/DOCX 、
JPG/PNG 等），支持导出评估报告为 PDF 或 Word 格式。

2.3 设计原则
     先进性 (Advancement)：系统深度整合了当前人工智能领域的前沿技术，包
括科大讯飞星火认知大模型、高精度 OCR、实时语音识别及人脸属性分析等。
后端采用主流的 Spring Boot 框架，前端采用 Vue.js 进行组件化开发，整体技
术架构具备行业先进性，确保了系统功能的智能化与高效性。
     安全性 (Security)：考虑到系统处理用户简历、面试音视频等高度敏感的个
人数据，安全性是设计的重中之重。系统将通过 API 密钥、访问令牌（Token）

                                    第 6 页 共 44 页
等机制进行严格的身份验证与权限控制。所有数据在网络中将采用 HTTPS 加密
传输，防止数据泄露与篡改，保障用户隐私安全。
  实用性 (Practicality)：系统功能设计紧密贴合高校学生的实际面试需求。
提供文字、语音、视频三种模式，满足不同场景下的训练要求。独创的六维能力
评分模型和个性化评测报告，为用户提供直观、可行的改进建议。采用异步处理
机制优化视频分析等耗时操作，保障了界面的流畅与响应速度，提升了用户体验。
  易维护性 (Maintainability)：清晰的模块划分和统一的 API 接口规范，降低
了系统的复杂度和维护成本。开发人员可以独立地对前后端或特定功能模块进行
升级和维护，而不会影响到整个系统的稳定运行。同时，完善的日志记录机制为
故障排查和系统监控提供了便利。

2.4 设计思想
  高内聚，低耦合 (High Cohesion, Low Coupling)：这是系统架构的核心思想。
前端 UI 层、后端业务逻辑层、第三方 AI 服务层之间通过明确的 API 接口进行
通信，各自独立。例如，评测报告的生成逻辑被内聚在后端，而人脸分析的具体
实现则由科大讯飞服务处理，后端仅负责调用和整合结果。这种设计使得各部分
可以独立开发、测试和部署，极大地提高了开发效率和系统的灵活性。
  面向服务 (Service-Oriented)：系统将核心的 AI 能力（如简历 OCR、大模
型评测、人脸分析）封装为独立的服务。这种面向服务的思想，使得这些能力不
仅可以在系统内部复用，也为未来将它们提供给其他校园应用（如职业规划系统）
创造了可能。
  异步化处理 (Asynchronous Processing)：对于视频处理、大模型分析等可能
耗时较长的任务，系统采用 Spring @Async 等技术进行异步化处理。用户提交面
试后可以立即获得初步响应，系统在后台完成复杂的分析任务，并通过轮询或消
息通知等方式将最终结果推送给用户，有效避免了长时间的等待，显著提升了用
户体验。

3. 流程设计

3.1 核心业务流程

  3.1.1 文字面试

  文字面试流程主要面向习惯于文字表达的用户。用户选择岗位和题量后，系
统自动生成并展示面试题目，用户逐题输入并提交答案。全部作答完成后，系统
对答案进行智能评估，生成详细的评测报告和改进建议，帮助用户了解自身在专
业知识、表达能力等方面的表现。




  3.1.2 语音面试
  语音面试流程为用户提供了更贴近真实面试的体验。用户选择岗位和题量后，
系统展示题目，用户通过语音作答并提交。系统将语音内容转为文字，用户可确
认或修正识别结果。全部题目完成后，系统进行综合评估，输出评测报告和个性
化建议，提升用户的口头表达与应变能力。




               第 8 页 共 44 页
  3.1.3 视频面试
  视频面试流程模拟真实面试场景，考察用户的综合素质。用户选择岗位和题
量后，系统展示题目，用户通过视频录制作答。系统对视频内容进行语音识别和
表情分析，并展示识别文本供用户确认。全部作答结束后，系统生成包含语言、
表情等多维度的评测报告，帮助用户全面提升面试表现。

3.2 简历分析
  简历分析流程支持用户上传个人简历（如 PDF、Word 等格式），系统通过
OCR 和自然语言处理技术自动提取简历中的关键信息，并进行智能分析。分析

  智能评测与报告生成流程在用户完成面试答题后自动启动。系统收集用户的
所有作答内容（包括文字、语音、视频等），调用大语言模型和相关 AI 服务进
行多维度评估。系统根据评估结果生成详细的面试报告，内容涵盖能力评分、优

9. 安全性设计

9.1 数据传输安全

      HTTPS 加密传输

   系统所有前后端通信、后端与科大讯飞 API 的交互均采用 HTTPS 协议，保
证数据在传输过程中经过 SSL/TLS 加密，防止数据被窃听、篡改或中间人攻击。
系统调用星火大模型 API 时，使用的地址为 https://spark-api-open.xf-yun.com/v2/
chat/completions，已强制 HTTPS。前端部署时，需在 Nginx 或 Tomcat 等服务器
上配置 SSL 证书，后端 Spring Boot 服务也应开启 HTTPS 端口，确保所有 API
请求均通过加密通道传输。对于未加密的 HTTP 请求，建议在服务器层做自动跳
转到 HTTPS，防止用户误用明文通道。

      敏感数据加密

   系统涉及的敏感数据包括用户身份信息、面试答案、简历文件、音视频数据
等。在数据传输过程中，这些内容均通过 HTTPS 加密通道传递，防止被第三方
窃取或篡改。
   在前端代码（如 Vue 页面）中，简历、视频、音频等文件通过 FormData 对
象以 multipart/form-data 方式上传，后端接口如/api/interview/upload-resume、/api
/interview/submit/video 等均采用 POST 请求，避免敏感数据暴露在 URL 中。
后端在接收文件时，需对文件类型、大小进行严格校验（如只允许 jpg/png/pdf
等格式，限制最大文件大小），防止恶意文件上传。

      防止重放攻击

   为防止数据包被截获后重复提交，需要在关键接口（如面试提交、简历上传、
答案提交等）增加时间戳、签名等机制。
   前端每次请求时，生成当前时间戳和随机数，并用用户 Token、请求体等信
息生成签名（如 HMAC-SHA256）。后端收到请求后，校验时间戳是否在允许范
围内（如 5 分钟内），并验证签名的正确性，防止请求被篡改或重放。对于上传
接口，在请求头或请求体中增加 nonce 和 timestamp 字段，后端记录已处理的请
求，避免重复处理。

9.2 API 访问安全

     身份认证与鉴权

   系统采用基于 Token 的用户身份认证机制。用户登录后，前端会将后端返回
的 Token（如 JWT）存储在本地，每次调用敏感 API 时自动携带 Token。后端通
过拦截器统一校验 Token 有效性，未认证用户无法访问核心接口。面试发起、答
案提交、简历上传等接口均需校验用户身份。

     接口权限控制

   后端接口根据用户身份和角色进行权限校验。普通用户只能访问和操作自己
的面试数据，管理员可管理所有用户和题库。以 /api/interview/complete/{intervie
wId}为例，后端会校验当前用户是否有权操作该面试记录，防止越权访问。

     防止接口滥用

   系统对外 API 设置了访问频率限制（Rate Limit），如同一用户或 IP 单位时
间内请求次数超限将被暂时封禁。上传接口（如简历、视频）已实现文件类型和
大小校验，防止恶意上传。

     防止 SQL 注入与 XSS

   后端所有数据库操作均采用 MyBatis Plus 等 ORM 框架，自动进行参数预编
译，防止 SQL 注入。前端输入内容在提交前进行转义，后端也会对用户输入做
安全校验，防止 XSS 攻击。

9.3 数据存储安全

     数据库访问安全

   后端采用 MySQL 8.0 数据库，数据库连接信息（如用户名、密码）统一配
置在 application.yml 文件中，未暴露在前端。数据库仅允许后端服务器访问，避
免外部直接连接，降低数据泄露风险。

     数据表结构安全

   所有核心数据表（如 interview_record、answer_record、ability_metric 等）均
设置主键、外键约束，保证数据完整性和一致性。例如，face_analysis 表中的 in
terview_record_id 为外键，级联删除，防止孤立数据。

11. 缺陷分析和系统优化方案

11.1 当前系统局限性分析

     异常处理与用户体验有待提升

  虽然系统在主要流程中实现了基础的异常捕获和错误提示，例如在语音识别、
简历上传、视频转写等环节能够捕获常见异常并返回错误信息，但整体补救措施
仍较为简单。当前大多数情况下，系统仅提示用户“请重试”或“请检查网络”，缺
乏更智能的引导。例如，用户多次重试无果时，系统未能自动分析错误原因、提
供具体操作建议（如切换网络、检查设备权限、联系客服等），也没有自动重试、
降级处理或缓存机制。部分异常仅在后台日志中记录，前端用户无法及时获知详
细原因，导致用户体验不佳，易产生困惑和流失。

     评测结果的准确性和个性化有限

  目前系统的能力评分和面试反馈主要依赖于通用大模型的输出，缺乏针对不
同行业、岗位、用户背景的深度定制。大模型虽然具备一定的通用性，但在专业
性、针对性和个性化方面存在不足。例如，针对技术岗与管理岗、应届生与有经
验者，系统输出的评测维度和建议大多相似，无法充分体现岗位差异和用户个性。
缺乏行业专家知识库、岗位能力画像等支撑，导致评测结果的专业性和说服力有
限，难以满足企业和个人用户对精准评测的需求。

     多端兼容性和适配性不足

  当前系统主要针对 PC 端 Web 浏览器进行开发，移动端适配和响应式设计不
完善，页面在手机、平板等设备上显示和交互体验较差。部分功能（如视频录制、
语音识别）在移动端兼容性不足，无法覆盖更多使用场景。系统未开发小程序、
App 等多端入口，限制了用户的访问方式和灵活性，影响了产品的市场覆盖面和
竞争力。
